# Codebase Cleanup Sweep — 2026-07-11

**Scope:** all of `app/src/main` (~14k lines Kotlin, ~100 files), reviewed in five parallel
subsystem passes (ViewModel/state, Compose screens, PDF composers, model/geom/io/util,
screen renderer/nav), each cross-checked against `docs/deep_audit_2026-07.md` so nothing
already remediated is re-reported. The five highest-stakes findings below were
independently re-verified against the code before writing this report (marked ✓).

This is a *findings* document — nothing here has been changed yet. Suggested sequencing
is at the bottom.

## Part 1 — Live bugs found by the sweep (not in the July audit)

### B-1. Multi-step redo is broken — first redo wipes the rest of the redo stack ✓
`ShaftViewModel.kt:2027-2039` (`redoLastDelete`) delegates to the public
`removeBody/removeTaper/...` APIs, and every `removeX` unconditionally runs
`redoHistory.clear()` (e.g. `ShaftViewModel.kt:920`, comment "new branch"). Delete A, B, C →
undo ×3 → redo once: A is re-deleted, and the pending redos for B and C are silently
destroyed. Both history deques are sized `MAX_DELETE_HISTORY = 10`, so multi-step redo is
clearly the intended design.
**Fix:** route redo through a private remove that takes `clearRedo: Boolean` (or an
`isRedoing` flag); pairs naturally with C-4 below.

### B-2. `rememberSaveable` holds an `ImageBitmap` — latent crash on backgrounding ✓
`RunoutRoute.kt:130` and `WearRoute.kt:98` keep the rendered preview bitmap in
`rememberSaveable`. `AndroidImageBitmap` is neither Parcelable nor Serializable, so
backgrounding the app (Home, recents, phone call) with a preview open throws during
`onSaveInstanceState`; even if it were saveable, a full-page ARGB bitmap (~7.8 MB) exceeds
the 1 MB Binder limit. The preview is regenerated by `LaunchedEffect` anyway.
**Fix:** plain `remember` for `previewBitmap` and `previewLoading` in both routes; only
`showPreview` deserves `rememberSaveable`.

### B-3. Runout/Wear PDF *export* ignores the prefs the *preview* uses ✓
The SAF export launchers (`RunoutRoute.kt:143-149`, `WearRoute.kt:110-116`) call
`composeRunoutPdf`/`composeWearPdf` without `pdfPrefs` or `lineThicknessScale`, so the
written file falls back to `PdfPrefs()` defaults while the on-screen preview
(`renderRunoutBitmap`/`renderWearBitmap`) passes `vm.currentPdfPrefs` and the thickness
snapshot. User sees a shaded, thickness-scaled preview and exports an unshaded,
default-weight PDF.
**Fix:** pass `pdfPrefs = vm.currentPdfPrefs, lineThicknessScale = ...` at both export
call sites (one-line each).

### B-4. Autosave snapshot drops state the .shaft format persists
`AutosaveManager.kt:30-38` (`SessionSnapshot`) omits `runoutConfig` (bubble overrides +
TIR direction), `unitLocked`, and `overallIsManual`. After process death: runout overrides
vanish; the `defaultUnitFlow` collector (`ShaftViewModel.kt:536-543`) may overwrite the
restored draft's unit because `unitLocked` restored as `false`; a manually-set OAL reverts
to auto and can be resized by the next `ensureOverall()`. Related: `importJson`
(`ShaftViewModel.kt:1665-1686`) never resets `_overallIsManual`, so the flag leaks between
opened documents.
**Fix:** add the three fields (defaulted; `ignoreUnknownKeys` keeps old drafts loadable),
restore them in `restoreSnapshot`, reset `_overallIsManual` in `importJson`.

### B-5. PDF compression note can disagree with the drawn geometry
`ShaftPdfComposer.kt:330` evaluates `showCompressionNote` against raw `spec.bodies`, but
the geometry is drawn from `bodiesForPdf` (resolved incl. auto-bodies, line 109). A long
auto-body draws a center break with no "compressed for clarity" note, and vice versa.
**Fix:** evaluate the note against `bodiesForPdf`.

## Part 2 — Ranked cleanup themes

### C-1. One shared profile renderer for the three PDF composers (large win)
The runout and wear profile renderers are the same ~100-line function twice
(`RunoutPdfComposer.kt:503-580` ≡ `WearPdfComposer.kt:226-329` — shade preamble, liner
ticks, thread hatch, slot call are line-for-line identical; only real deltas are the
body-loop factoring and that wear silently omits taper keyways runout draws — an
undocumented divergence). The body/taper/liner primitives exist in a *third* copy in
`ShaftPdfComposer.kt` (bodies 578-628, tapers 720-753, liners 909-929), and the
compression constants (`COMPRESS_TRIGGER_PT = 220f`, `ZIGZAG_GAP_MAX_PT = 20f`) are
declared privately in all three files — Runout's even carries the comment "(matches
ShaftPdfComposer threshold)", a hand-maintained invariant. Headers and the arrowed OAL
line are also duplicated pairwise (`RunoutPdfComposer.kt:412-487` vs
`WearPdfComposer.kt:149-220`), and the `shadeFill()` recipe `argb(40,0,0,0)` exists 3×.
**Fix:** extend the `BreakSymbol.kt` pattern — a shared `drawProfile(c, spec, style)` with
a `ProfileStyle` (paints + `drawKeyways` flag), shared `drawJobInfoLine`/`drawOalArrowLine`
helpers, and the compression constants moved to `BreakSymbol.kt` as `internal`. The
keyway divergence becomes an explicit flag instead of an accident.

### C-2. Dead subsystems: delete or wire them (large win, mostly deletion)
- **Cross-type component order is write-only** ✓ — `ComponentCarouselPager` declares
  `componentOrder` (`ComponentCarousel.kt:111`) and never reads it; pages order by
  `resolvedComponents`. Everything feeding it (`orderAdd`, `orderRemove`,
  `ensureOrderCoversSpec`, `beforeOrder` undo snapshots, `moveComponentUp/Down`,
  `snapChainFrom*` — `ShaftViewModel.kt:394-395, 789-797, 1807-1872`) is maintenance of
  state nobody observes, running on every add/remove/undo/import.
  **Decide the contract first**: `ComponentsOrdering.md` claims newest-on-top is "LOCKED" —
  either wire `componentOrder` into the pager or delete the subsystem and update the docs.
- **`ui/theme` package is never applied** ✓ — no file imports it; `MainActivity.kt:34`
  wraps content in bare `MaterialTheme {}`, so no dark theme, no dynamic color, and the
  custom Typography never applies. Screens read `MaterialTheme.colorScheme` believing a
  theme is in effect. **Wiring `ShaftSchematicTheme` in is a one-line, user-visible win**
  (verify preview colors in dark mode) — or delete the three files and document baseline
  theme as intentional.
- **Renderer options/plumbing:** ~19 of ~40 `RenderOptions` fields are never read
  (`RenderOptions.kt:25-67`; grid fields actively mislead — `GridRenderer` hardcodes its
  own values); highlight edge-ring params are threaded through three files for a feature
  the comments say was removed (`ShaftRenderer.kt:118-122, 576-613`); the
  `ThreadStyle.UNIFIED` path (~60 lines) and the `textMeasurer` param are unreachable;
  `ReferenceEnd` is defined twice and both are dead.
- **`AddDefaultsConfig`:** all ten `default*Mm(unit)` helpers have zero callers, and every
  branch is a no-op (MM constants are exactly IN × 25.4). Real defaulting lives in
  `SessionAddDefaults.initial()`. Delete helpers + MM twins; `docs/Defaults.md` points at
  the dead API and needs updating.
- **Three dead model files:** `RendererAliases.kt`, `TaperDirection.kt` (shadowed by a
  private enum in `TaperTitles.kt`), `ShaftSpecMigrations.kt` (whose private
  `String?.isBlank()` shadows stdlib with *different* semantics — a trap if revived).
- **Dead VM API:** `newShaft()` (`ShaftViewModel.kt:1597-1614`) — zero callers and, unlike
  `newDocument()`, doesn't reset `_currentDocumentName`, so a future caller could silently
  overwrite the previous document's quick-save file; `didRestoreAutosave` pair; the
  `unlockAchievement(Definition)` overload; `computeBodyOnlyPtPerMm` +
  `computePdfPtPerMmFitAxes` + `fmtLen` + `hasAftTaper`/`hasFwdTaper`
  (`ShaftPdfComposer.kt:508-542, 1116-1179`, ~60 lines; `BodyOnlyScalingTest` exercises a
  function production no longer runs — retarget it at `computeDetailPtPerMm`).
- **Dead screen-layer items:** `ComponentType.kt` (whole file), `ExpandableSection` +
  `clickableWithoutRipple`, write-only `highlightId` (`ShaftScreen.kt:289`), dead callback
  params threaded through 4 layers (`onSetUnit`, `onToggleGrid`, `unitLocked`,
  `onMoveComponentUp/Down`, `onNavigateHome`, `onExportRunout/Wear`,
  `onUpdateCouplerBoltSlotLabel` — the slot card has no title editor), ~16 unused imports
  in ShaftScreen.kt; `DimSpan.labelBottom` + renderer branch (never non-null in
  production); `PdfDimensionRenderer` dead ctor params + empty init; assorted flotsam
  (`filterDecimalPermissive`, `SHAFT_MIME`, `normalizeJsonName`, `findRightNeighbor`,
  `Threads.hasPitch`, `Liner.startMmPhysical`, `textSmall`, unused `spec` params and the
  stale "OAL is included in the header" KDoc in `RunoutPdfComposer.kt:407-430`).

### C-3. Preview renderer hot path (medium win, cheap fixes)
The draw lambda reads `offset.value`/`scale.value`, so during pan/zoom it runs per frame:
- `layout.dbg()` is called **unconditionally** before the verbose-log check
  (`ShaftDrawing.kt:291`) — six `String.format`s per frame with logging off. One-line fix.
- `ShaftLayout.compute` depends only on (spec, resolved, size) but runs per frame
  (`ShaftDrawing.kt:281`); cache via `drawWithCache` or remembered holder.
- Per-frame allocations in `ShaftRenderer.draw`: four `filterIsInstance` lists, a
  `Stroke` per outlined shape, a `Path()` per taper, an O(tapers²) id lookup
  (`ShaftRenderer.kt:124-448`). The file's own header says "Keep allocations low."
- `RenderOptions` (40 fields) rebuilt every recomposition (`ShaftDrawing.kt:141-172`);
  RunoutRoute already shows the remembered pattern.
- ~230 lines duplicated between resolved-vs-spec draw branches (`ShaftRenderer.kt:129-448`)
  — normalize inputs at the top and keep one loop per type; cuts the file in half.
- `Layout` is a field-for-field copy of `ShaftLayout.Result` with a per-draw adapter
  (`ShaftRenderer.kt:55-79`); delete it.

### C-4. ViewModel duplication (medium win; unlocks B-1's clean fix)
- Five near-identical ~50-line `removeX` functions (`ShaftViewModel.kt:878-1573`) — extract
  one `removeComponent(id, kind, ...)` + shared `recordDelete` tail; the redo fix (B-1)
  then lands in one place.
- Five character-identical label updaters + three reference updaters (~110 lines,
  `ShaftViewModel.kt:1042-1533`) — one generic `updateAt` helper.
- `LastDeleted` sealed hierarchy: `listIndex`/`orderIndex` are write-only; collapse to one
  data class with `id/kind/beforeSpec/beforeOrder` (`ShaftViewModel.kt:54-115`).
- Resolved-components pipeline runs synchronously on `Main.immediate` per keystroke
  (`ShaftViewModel.kt:484-490`); replace collector with
  `combine(...).flowOn(Default).stateIn(...)` — same semantics, off-main. (The
  per-keystroke OAL commit contract is untouched.)
- OAL "revert to Auto" transition is triplicated in ShaftScreen callbacks
  (`ShaftScreen.kt:578-651`) and belongs in the VM as one `setOverallAuto()`.

### C-5. Shared controls for carousel cards and Add dialogs (medium win, strengthens parity)
The parity rule ("dialogs mirror cards") is currently upheld by hand-copying:
- Editable-title block ×4 in the carousel (~120 lines, `ComponentCarousel.kt:407-776`);
  the slot card drifted and has none.
- AFT/FWD chip row ×4 in the carousel + a *visually different* `DirectionChip` in the
  dialogs (`AddComponentDialogs.kt:674-688`) — five copies, two designs, hardcoded
  black/white ignoring theme.
- "Add Anyway?" warning dialog + state ×3 (`AddComponentDialogs.kt:186-566`).
- A third numeric widget: dialogs use raw `CommitNumField` bypassing
  `NumericInputField`'s filter contract (its header says all numeric fields must use it),
  and it triple-commits (keystroke + onDone + blur).
- Two divergent "defaults for a new component" computations (`ShaftScreen.kt:1460` vs
  `AddComponentDialogs.kt:65-83`).
**Fix:** one shared file with `EditableTitle`, `AftFwdSelector`, `AddWarningGate`,
`computeAddDefaults`; dialogs and cards compose the same controls — the invariant becomes
structural instead of manual.

### C-6. Route-side PDF plumbing (small-medium win)
- The temp-file → `PdfDocument` → `PdfRenderer` → bitmap pipeline exists 3×
  (`RunoutRoute.kt:530-576`, `WearRoute.kt:240-285`, `PdfPreviewScreen.kt:323-383`), as do
  `openXxxPdf` and filename builders. Extract `renderPdfPageToBitmap` +
  `openExportedPdf` + `buildExportFilename`.
- PDF options controls exist in three parallel implementations that already disagree
  (`PdfOptionsSheet`, `RunoutWearOptionsSheet`, Settings' PDF page); both sheets take the
  whole ViewModel. Extract `PdfShadeToggles` / `LineThicknessSlider` / `TieringModeSelector`.
- The thickness slider writes DataStore **and** re-renders a full PDF preview on every
  drag frame (3 screens); commit on `onValueChangeFinished` or debounce.
- Eleven copy-pasted switch rows in `DeveloperOptionsRoute` (+ more in Settings) → one
  `ToggleRow` with `Modifier.toggleable` (also fixes tap-target accessibility).
- `HorizontalPager` lacks a `key` (`ComponentCarousel.kt:234`) — page state is positional;
  deleting page 2 hands its scroll state to the next component. Add
  `key = { component id }`.

### C-7. Model-layer consolidation (small-medium win)
- "Furthest occupied end" implemented 4× — `coverageEndMm`, `lastOccupiedEndMm`, a private
  VM copy, and a ThreadTitles local that *includes* excluded threads (same name, different
  semantics). Keep `coverageEndMm()`; rename the ThreadTitles variant honestly.
- Open-interval overlap predicate re-declared 6× (model/ui.util/resolved/geom); one
  `spansOverlap(...)` next to `Segment`.
- `withSegmentStart`: five identical 9-line branches — add `Segment.withStart()` and
  collapse.
- Title builders (Taper/Thread/Liner) re-implement the same group→count→number algorithm
  3× (~200 lines → ~80); the four `ui/util/*Naming.kt` files are pure delegation shims
  creating two import paths for one function — delete them.
- End-feature detection duplicated between `ShaftPdfComposer.kt:1213-1316` and
  `geom/OalComputations.kt:14-91` with different tolerances (0.5 mm vs 1e-3 mm) — the
  footer and SET positions can classify the same shaft differently. One parameterized API
  in `geom`.
- Unit-conversion sprawl: `UnitSystem.toMillimeters/fromMillimeters` are production-dead
  while 25.4 is declared six-plus times (one in the *model* package). One constant, one
  converter.
- `computeOalWindow` is now an identity window (always `(0, OAL)`), but ShaftScreen still
  memoizes it keyed on threads/tapers and builds `spec.copy(...)` round-trips through it —
  collapse call sites or fix the stale keys.
- `resolveComponents` sorts 3× per resolution (once provably wasted); runs per keystroke.
- Save-path filename sanitization gap: Rename strips illegal characters
  (`InternalDocRoutes.kt:82-90`), Save does not (`:584`) — move sanitization into
  `InternalStorage.normalizeShaftDocName` (the shared choke point) + unit test.
- Small stuff: `ComponentWarnings`' `Float.MIN_VALUE..` idiom ×4 → one `Segment` helper;
  stale file headers (ShaftSpecExtensions, ComponentOrder KDoc missing slots); dead elvis
  in `Threads.normalized()`; `updateTaper`'s redundant self-copies
  (`ShaftViewModel.kt:1002-1006`); `composeShaftPdf` re-computes `computeOalWindow` 3× and
  `computeSetPositionsInMeasureSpace` 2× per export — compute once, pass down.
